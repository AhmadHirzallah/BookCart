# The Single-Value Value Object Base — Killing the `Value` + `Create` Redundancy

> **Companion to:** _"Rich Domain Model & Factory Pattern — The BookCart Playbook"_ and
> _"Strongly-Typed IDs & the Abstract Base Pattern"_.
>
> You already applied this exact move once — `AStronglyTypedId` absorbed the repeated
> `Value` / `ToString()` out of every id. This guide does the same for **single-value Value
> Objects** (`CategoryName`, `CategoryDescription`, a future `Isbn`, `Slug`, `Price`, …) and,
> crucially, teaches you **where the abstraction must STOP** — because that line is what
> separates a senior design from an over-engineered one.

---

## Part 0 — Name the problem precisely

Look at what repeats across `CategoryName`, `CategoryDescription`, and every future wrapper:

```csharp
public sealed record CategoryName
{
    public string NameValue { get; private set; }          // ← (1) a single backing value
    private CategoryName(string v) { NameValue = v; }       // ← (2) private ctor
    public static Result<CategoryName> Create(string v)     // ← (3) a Create factory returning Result
    {
        var errors = new List<Error>();                     // ← (4) the SAME plumbing
        /* rules */                                          // ← (5) the ONLY part that differs
        if (errors.Any()) return errors;                    // ← (4) the SAME plumbing
        return new CategoryName(v);                          // ← (4) the SAME plumbing
    }
    // ToString(), equality, GetHashCode …                  // ← (1) the SAME mechanics
}
```

Split every line above into two buckets — this is the whole guide in one table:

| Bucket | Lines | Varies per VO? | Where it belongs |
|--------|-------|----------------|------------------|
| **MECHANICS** — storing the value, `ToString()`, equality, the *shape* of `Create` | (1)(2)(4) | ❌ Never | **Centralize** in an abstract base |
| **POLICY** — the actual validation rules, the max length, which errors | (3)(5) | ✅ Always | **Keep local** in each `sealed record` |

> **The senior insight (memorise it):**
> **Centralize mechanics. Localize policy.**
> DRY applies to *how a VO works* (identical everywhere). It must **not** be applied to *what a
> VO means* (different everywhere) — forcing rules into a base is the classic over-abstraction
> that creates a god-class the whole domain must fight.

This is the same reason `AStronglyTypedId` centralizes `CheckFormat` (mechanic) but returns a
`FormatCheck` reason and lets each id choose its own `Error` (policy). You already know this shape.

## Part 1 — Why a Value Object is a `record` in the first place (foundation)

Before the base class, be crystal-clear on *why* these are `record` and not `class` — it's the
reason the whole pattern works.

A **Value Object has no identity**. Two `CategoryName("Fiction")` are — by definition —
**the same thing**. Contrast an *entity*: two `Category` rows both named "Fiction" are *different*
categories (different `Id`). So:

| | Entity (`Category`) | Value Object (`CategoryName`) |
|---|---|---|
| Equality is by… | **identity** (`Id`) | **value** (the wrapped data) |
| `new X("Fiction") == new X("Fiction")` | `false` (different rows) | **`true`** (same value) |
| C# construct | `class` (reference equality, override `Equals` on Id) | **`record`** (structural equality for free) |
| Mutability | mutable via guarded methods | **immutable** (a new value = a new object) |

**`record` gives you, generated by the compiler:** value-based `Equals`, `GetHashCode`, `==`/`!=`,
and a `with` expression — *exactly* the semantics a Value Object needs. Writing this by hand on a
`class` is ~30 lines of error-prone boilerplate. **That is why every VO in this codebase is a
`record`.**

## Part 2 — The solution: `ASingleValueObject<TValue>`

Put this next to your other abstractions (`BookCart.Domain/Common/Abstractions/`), beside
`AStronglyTypedId`. It is deliberately tiny — it owns **only** the mechanics.

```csharp
namespace BookCart.Domain.Common.Abstractions;

/*
 *? The shared base for every SINGLE-VALUE Value Object — a VO that wraps exactly one primitive
 *? (CategoryName→string, Price→decimal, Quantity→int, Isbn→string, …).
 *
 *? What lives here (MECHANICS — identical for every single-value VO):
 *?   - Value storage (generic, so string/int/decimal all reuse it)
 *?   - ToString()  (sealed — see the deep-dive on why)
 *?   - equality / GetHashCode / == / !=  (inherited from 'record', compares Value + type)
 *
 *? What deliberately does NOT live here (POLICY — different for every VO):
 *?   - the validation rules            (non-empty? max length? positive? range?)
 *?   - which Error to raise            each VO owns its rules and its errors
 *?   - the public Create factory       each VO exposes its own, calling its own private ctor
 *
 *! abstract  → you can never 'new' a bare value object; only a concrete VO is real.
 *! TValue : notnull → the wrapped value is never null; "no value" is modelled by a nullable
 *!                    PROPERTY on the entity (Description?), not by a null inside the VO.
 */
public abstract record ASingleValueObject<TValue>
    where TValue : notnull
{
    #region Value + Construction

    //? The single wrapped, already-validated primitive. One canonical name for ALL VOs.
    public TValue Value { get; }

    //! protected: a value can only be set by a concrete VO's private ctor → which is only
    //! reachable from that VO's Create factory → which validated first. Invalid state = impossible.
    protected ASingleValueObject(TValue value)
    {
        Value = value;
    }

    #endregion

    #region ToString

    /*
     *! sealed — same load-bearing reason as AStronglyTypedId:
     *! every 'record' auto-generates ToString(); without 'sealed' each concrete VO would
     *! regenerate it and print "CategoryName { Value = Fiction }". 'sealed override' forces
     *! ONE ToString for the whole hierarchy → the raw value, clean in logs and serialisation.
     */
    public sealed override string ToString() => Value.ToString() ?? string.Empty;

    #endregion
}
```

That is the entire reusable core. Notice it references **no** `Error`, no `CategoryErrors`,
nothing concrete — it is a *leaf* in the dependency graph, exactly like a good base should be (DIP).

## Part 3 — The concrete VOs shrink to just their policy

### `CategoryName` — refactored

```csharp
using BookCart.Domain.Common.Abstractions;
using BookCart.Domain.Common.Result;
using BookCart.Domain.Common.Result.Errors;

namespace BookCart.Domain.Entities.Categories.ValueObjects;

public sealed record CategoryName : ASingleValueObject<string>
{
    public const int MaxLength = 1000;

    //! private → the ONLY caller is Create(); guarantees the string was validated + trimmed first.
    private CategoryName(string value) : base(value) { }

    public static Result<CategoryName> Create(string? value)
    {
        var errors = new List<Error>();

        if (string.IsNullOrWhiteSpace(value))
        {
            errors.Add(CategoryNameErrors.Required);
        }
        else if (value.Trim().Length > MaxLength)
        {
            errors.Add(CategoryNameErrors.TooLong(MaxLength));
        }

        if (errors.Count > 0)
        {
            return errors;   //! implicit: List<Error> → Result<CategoryName> (failure). See §4.5.
        }

        return new CategoryName(value!.Trim());   //! implicit: CategoryName → Result<CategoryName> (success)
    }
}
```

**Everything gone vs your original:** no `NameValue` property, no `ToString`, no equality, no second
constructor. What remains is *only* `CategoryName`'s reason to exist — its rules. That is SRP made
physical: **one file, one responsibility, one reason to change.**

### `CategoryDescription` — the new one, now trivial

```csharp
using BookCart.Domain.Common.Abstractions;
using BookCart.Domain.Common.Result;
using BookCart.Domain.Common.Result.Errors;

namespace BookCart.Domain.Entities.Categories.ValueObjects;

public sealed record CategoryDescription : ASingleValueObject<string>
{
    public const int MaxLength = 4000;

    private CategoryDescription(string value) : base(value) { }

    public static Result<CategoryDescription> Create(string? value)
    {
        var errors = new List<Error>();

        if (string.IsNullOrWhiteSpace(value))
        {
            errors.Add(CategoryDescriptionErrors.Required);
        }
        else if (value.Trim().Length > MaxLength)
        {
            errors.Add(CategoryDescriptionErrors.TooLong(MaxLength));
        }

        if (errors.Count > 0)
        {
            return errors;
        }

        return new CategoryDescription(value!.Trim());
    }
}
```

> **On "optional" VOs:** don't model *optional* by allowing `null` **inside** the VO. A
> `CategoryDescription` that exists is always valid & non-empty. "The category has no description"
> is expressed on the **entity** as `public CategoryDescription? Description { get; private set; }`
> — a nullable *property*, not a null *value*. The VO stays a guarantee; the optionality lives one
> level up. (This is why the base is `TValue : notnull`.)

## Part 4 — Deep dives (the "why", and what breaks)

### 4.1 — Why `abstract` on the base, `sealed` on every concrete VO?

This is the exact question you asked. Two opposite modifiers, two precise reasons:

**`abstract record ASingleValueObject<TValue>`**
- `abstract` = *"this is an incomplete concept; you cannot instantiate it directly."* A bare
  "single value object" is meaningless — only a *CategoryName* or a *Price* is real. `abstract`
  makes `new ASingleValueObject<string>("x")` a **compile error**, which is correct.
- It **must** be inheritable (not sealed) — the whole point is that concrete VOs derive from it.

**`sealed record CategoryName`**
- `sealed` = *"nothing may inherit from this."* For a Value Object this is not a nicety — it
  **protects the equality invariant** and Liskov substitution:
  - Imagine `record TrimmedName : CategoryName` adding a field. Now `CategoryName("a")` and
    `TrimmedName("a")` have different `EqualityContract`, so they're **unequal even with the same
    value** — silently breaking the one rule a VO must uphold ("same value ⇒ equal"). Records
    fold the runtime type into equality precisely to stop cross-type false-positives; sealing stops
    you from *creating* the confusing subtype in the first place.
  - A Value Object is a **final, whole concept**. There is no "kind of CategoryName." Sealing
    encodes that truth and lets the JIT devirtualize calls (a minor perf bonus).

> **The rule, portable to any hierarchy:** *the shared base is `abstract` (extensible, never
> instantiated); the domain leaf is `sealed` (final, never extended).* You applied it already:
> `AStronglyTypedId` (abstract) → `CategoryId` (sealed). Same law here.

### 4.2 — Why generic `<TValue>` and not just `string`?

Because the redundancy isn't string-specific — you'll wrap `decimal Price`, `int StockQuantity`,
`DateOnly PublishedOn`. A generic base means **one** mechanic-holder serves every primitive:

```csharp
public sealed record Price          : ASingleValueObject<decimal> { /* rules: >= 0 */ }
public sealed record StockQuantity  : ASingleValueObject<int>     { /* rules: >= 0 */ }
public sealed record Isbn           : ASingleValueObject<string>  { /* rules: 13 digits */ }
```

`where TValue : notnull` bans `ASingleValueObject<string?>`, keeping the "the value is always
present and valid" guarantee airtight.

### 4.3 — Why NOT push `Create` into the base too? (the line you must not cross)

Tempting — but `Create` contains the **rules**, and rules differ. Three forces stop you:

1. **SRP.** If the base held the rules, it would change every time *any* VO's rule changed →
   a magnet for edits, the opposite of stable.
2. **OCP.** Adding `Price` should require **zero** edits to the base. A rules-aware base would
   need a new branch for every new VO — you'd be *modifying* to *extend*.
3. **The C# wall.** `static` methods aren't polymorphic, and the base cannot call a derived
   `private` constructor. So a base `Create` literally cannot build the right subtype without the
   derived type handing it a factory delegate — which brings us to the *optional* pattern below.

#### Optional: a template helper to shave the `Create` plumbing (advanced, use with judgement)

If the `new List<Error>() … if (Count>0) return … return new(...)` scaffolding genuinely bothers
you, you *can* centralize the **shape** (not the rules) via a template method that takes the rules
as a delegate:

```csharp
// add inside ASingleValueObject<TValue>
protected static Result<TSelf> Build<TSelf>(
    TValue value,
    Func<TValue, IReadOnlyList<Error>> validate,   // ← the caller supplies the POLICY
    Func<TValue, TSelf> factory)                    // ← and how to construct itself
    where TSelf : ASingleValueObject<TValue>
{
    IReadOnlyList<Error> errors = validate(value);
    return errors.Count > 0 ? errors.ToList() : factory(value);
}
```

```csharp
public sealed record CategoryName : ASingleValueObject<string>
{
    public const int MaxLength = 1000;
    private CategoryName(string value) : base(value) { }

    public static Result<CategoryName> Create(string? value) =>
        Build((value ?? string.Empty).Trim(), Validate, v => new CategoryName(v));

    private static IReadOnlyList<Error> Validate(string value) =>
        value.Length == 0 ? [CategoryNameErrors.Required]
      : value.Length > MaxLength ? [CategoryNameErrors.TooLong(MaxLength)]
      : [];
}
```

> **My recommendation:** start **without** `Build`. Ship the plain `Create` from Part 3. Only
> introduce the template once you have **three** VOs whose scaffolding is provably identical
> (the *Rule of Three*). Premature template-method here trades a few readable lines for indirection
> a newcomer must decode. Senior ≠ "most abstract"; senior = "abstract *at the right time*."

### 4.4 — Should a VO have the EF parameterless `private CategoryName() { }`?

**It depends on the mapping strategy — and usually NO for value converters:**
- **Value converter** (`CategoryName ⇄ string`, the common choice for single-value VOs): EF stores
  the raw `string` column and rebuilds the VO **through your converter expression**, *not* through
  a constructor. → **No parameterless ctor needed.** Your original `private CategoryName() {}` was
  unnecessary *and* triggered a nullable warning (`NameValue` left unassigned → `CS8618`).
- **Owned entity** (`OwnsOne`): EF materializes it like a mini-entity and *does* want a
  parameterless ctor. → include it **only then**.

Since single-value VOs map far more naturally as value converters, the Part 3 code omits the
parameterless ctor on purpose. (Contrast entities like `Category`, which **always** need the private
parameterless ctor — see the Rich Domain playbook §3.2.)

### 4.5 — The compiler error you hit, fully explained

```csharp
//! Cannot implicitly convert 'Result' to 'Result<CategoryName>'…
return Result<CategoryName>.Failure(errorsList);   // ❌
```

**Why it fails:** `Failure` is **not** a member declared on `Result<TValue>`. It lives on the
non-generic base `Result` in two forms — `Result Failure(IReadOnlyList<Error>)` (returns the
*non-generic* `Result`) and `Result<TValue> Failure<TValue>(IReadOnlyList<Error>)` (a **generic
method**). Writing `Result<CategoryName>.Failure(errorsList)` binds to the **non-generic** overload
(nothing tells it `TValue`), so it returns a base `Result`, which won't implicitly become
`Result<CategoryName>`. C# generic methods **cannot infer their type argument from the return/target
type** — only from arguments.

**Two correct fixes:**

```csharp
return errorsList;                              // ✅ cleanest — uses your implicit operator List<Error> → Result<TValue>
return Result.Failure<CategoryName>(errorsList); // ✅ explicit — call the GENERIC method, name TValue yourself
```

Prefer the first: your `Result` type *deliberately* defines
`implicit operator Result<TValue>(List<Error>)` so factories read cleanly. Lean on it.

### 4.6 — Meta-insight: your ID base is *also* a single-value VO

Notice `AStronglyTypedId` is, structurally, `ASingleValueObject<string>` **plus** id-specific
mechanics (`Prefix`, `New`, `From`, `CheckFormat`). You *could* express:

```csharp
public abstract record AStronglyTypedId : ASingleValueObject<string> { /* + id mechanics */ }
```

That's a legitimately elegant unification (an identity *is* a single-value VO). **But** your
`AStronglyTypedId` already works and is committed — refactor it only if you want the conceptual
tidiness, and do it as its own isolated change. Recognizing the shared abstraction is the mastery;
*rewriting working code to prove it* is optional.

## Part 5 — How this satisfies OOP / OOD / SOLID (say it in an interview)

| Principle | How this design honors it |
|-----------|---------------------------|
| **Encapsulation (OOP)** | `Value` is get-only; the only constructor is `private`; the only door is `Create`. An invalid VO is unconstructable. |
| **Abstraction (OOP)** | Callers think in `CategoryName`, never `string`. The primitive is hidden behind meaning. |
| **Inheritance (OOP)** | Used **for mechanics reuse only** (Value/ToString/equality), never to model an "is-a-kind-of" taxonomy of values. |
| **Polymorphism (OOP)** | Any `ASingleValueObject<T>` can be logged/serialized uniformly via the one `ToString`. |
| **SRP** | Each VO file has exactly one reason to change: *its own rule*. The base changes only if the *mechanics* of value-wrapping change (≈ never). |
| **OCP** | New VO (`Price`, `Isbn`) = new sealed record. **Zero** edits to the base. Open to extension, closed to modification. |
| **LSP** | Sealing the leaves + records' `EqualityContract` guarantee no subtype can violate "same value ⇒ equal." Every VO is safely substitutable for its base. |
| **ISP** | The base exposes a razor-thin surface (`Value`, `ToString`). No VO is forced to depend on members it doesn't use. |
| **DIP** | The base depends on **nothing concrete** (no `Error`, no `CategoryErrors`). Concrete VOs depend on the abstraction. Arrows point inward — Clean Architecture intact. |
| **DRY** | Applied to *mechanics* (centralized) — **not** to *policy* (deliberately duplicated because it isn't duplication, it's distinct rules). |
| **Cures: Primitive Obsession** | Strings/decimals never roam the domain naked; each is a validated, named type. |

## Part 6 — FAST CHECKLIST: adding any new single-value VO

```
NEW SINGLE-VALUE VALUE OBJECT — build order
───────────────────────────────────────────
1. Errors first     XxxErrors.cs → static readonly Error Required; static Error TooLong(int max)…
2. Declare          public sealed record Xxx : ASingleValueObject<TPrimitive>
3. Constant(s)      public const int MaxLength = …;   (limits live WITH the type)
4. Private ctor     private Xxx(TPrimitive value) : base(value) { }
5. Create()         public static Result<Xxx> Create(TPrimitive? input)
                       → new List<Error>(); apply RULES; if (Count>0) return errors;
                       → normalize (Trim); return new Xxx(clean);
6. (skip)           NO Value property, NO ToString, NO Equals — all inherited.
7. EF ctor?         only if you'll map it as an OWNED entity; value-converter mapping needs none.
```

## Part 7 — Cheat sheet & "what if I don't?"

```
DECISION DECODER
────────────────
record (not class)          → value equality for free; two equal values ARE equal   ← every VO
abstract record  (base)     → reusable mechanics; can NEVER be instantiated bare
sealed record    (leaf)     → final concept; protects the "same value ⇒ equal" invariant (LSP)
TValue : notnull            → the wrapped value is always present; optionality lives on the entity
Value  (base property)      → ONE canonical name for the payload across every VO
private ctor + static Create→ the only validated door in
return errorsList;          → implicit List<Error> → Result<T>  (do NOT write Result<T>.Failure(...))
```

| You do… | What breaks |
|---------|-------------|
| `class` instead of `record` | you hand-write `Equals`/`GetHashCode`/`==`; miss one → subtle equality bugs, broken dictionary keys |
| forget `sealed` on the leaf | a subtype can be added whose value-equality silently disagrees → LSP violation |
| forget `abstract` on the base | someone `new ASingleValueObject<string>("x")` — a meaningless, ruleless value |
| push rules into the base | base becomes a god-class edited on every VO change → SRP & OCP gone |
| allow `null` inside the VO | "valid VO" no longer means "has a value"; every consumer must re-null-check |
| `Result<T>.Failure(list)` | `CS0266: cannot convert 'Result' to 'Result<T>'` — use `return list;` or `Result.Failure<T>(list)` |
| add EF parameterless ctor for a value-converter VO | dead code + `CS8618` nullable warning on the unassigned value |
```
